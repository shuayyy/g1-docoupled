1.3 Fixed startup device parsing in _super_init
-----------------------------------------------
Problem:
The copied _super_init() in g1.py expected sim_device to be an object with a
.type field:

    if sim_device.type == 'cuda' and sim_params.use_gpu_pipeline:

But task_registry passes sim_device as a string like:
- "cuda:0"
- "cpu"

This caused:
AttributeError: 'str' object has no attribute 'type'

Fix:
Imported gymutil and used the standard Isaac Gym parsing path:

    from isaacgym import gymapi, gymtorch, gymutil

    sim_device_type, self.sim_device_id = gymutil.parse_device_str(self.sim_device)

    if sim_device_type == 'cuda' and sim_params.use_gpu_pipeline:
        self.device = self.sim_device
    else:
        self.device = 'cpu'

Why this mattered:
The rest of g1.py already assumes:
- self.sim_device_id exists,
- self.device is either a string "cpu" or a sim device string like "cuda:0".

So this one fix restored consistency with the rest of the file and with the
base task behavior.

15. Detailed explanation of the current trajectory blocker
----------------------------------------------------------

This is the current runtime blocker:

- [g1.py](/home/shuaiyyy/humanoids_repo/g1_docoupled/g1_decoupled/IL_transformer/legged_gym/legged_gym/envs/g1/g1.py#L143) loads `self.cfg.human.filename`
- [human.py](/home/shuaiyyy/humanoids_repo/g1_docoupled/g1_decoupled/IL_transformer/legged_gym/legged_gym/utils/human.py#L8) adds the loaded motion to the G1 default pose offset
- but the only motion file currently present is:
  [ACCAD_walk_10fps.npy](/home/shuaiyyy/humanoids_repo/g1_docoupled/g1_decoupled/IL_transformer/legged_gym/data/ACCAD_walk_10fps.npy)
- that file has shape:
  `(77, 19)`

15.1 What the code is trying to do
----------------------------------

The copied G1 environment assumes the trajectory file contains one joint target
vector per frame, and that the size of that vector matches the robot DOF count.

The relevant flow is:

1. In `g1.py`, `_init_target_jt()` calls:

       load_target_jt(self.device, self.cfg.human.filename, self.default_dof_pos)

2. In `human.py`, `load_target_jt()` does:

       one_target_jt = np.load(f"data/{file}").astype(np.float32)
       one_target_jt = torch.from_numpy(one_target_jt).to(device)
       target_jt = one_target_jt.unsqueeze(0)
       target_jt += offset

3. Here, `offset` is `self.default_dof_pos`, which in G1 has one value per DOF.

So the code expects:

- trajectory shape = `(num_frames, num_dofs)`
- offset shape = `(1, num_dofs)` after broadcasting

For your G1 config:
- `num_dofs = 29`

So the code expects:
- `(num_frames, 29)`

15.2 What is actually present
-----------------------------

The only available file is:
- `ACCAD_walk_10fps.npy`

Its shape is:
- `(77, 19)`

That means:
- 77 frames
- 19 joint values per frame

This is compatible with an H1-sized or H1-like target layout, not with G1’s
29-DOF layout.

15.3 Why the exact runtime error happens
----------------------------------------

When `human.py` executes:

    target_jt += offset

the tensor shapes are effectively:

- `target_jt`: `(1, 77, 19)`
- `offset`: `(1, 29)`

Broadcasting cannot align 19 with 29 on the joint dimension.

So PyTorch raises:

    RuntimeError: The size of tensor a (19) must match the size of tensor b (29)

This is not a random runtime bug. It is a direct statement that the motion data
and the robot joint layout do not match.

15.4 What this means semantically
---------------------------------

This tells us the copied environment logic is now functioning far enough to reach
the imitation data path, but the imitation data is still for a different robot
or a different joint set.

In practical terms:

- G1 environment side says:
  "I have 29 actuated joints and I want a 29-value target vector each frame."

- Motion file side says:
  "I only provide 19 target joint values per frame."

So the environment does not know how to map the 19-value motion into the 29-DOF
G1 body by itself.

15.5 Why this is the first real G1-specific blocker
---------------------------------------------------

Everything before this point was codebase plumbing:
- imports
- task registration
- device parsing
- terrain generation
- asset loading
- config field mismatches

This blocker is different.

It is the first place where the code is asking:

"What exact motion representation is G1 supposed to imitate?"

and the current data cannot answer that, because it is not in G1 joint space.

15.6 What does not solve this problem
-------------------------------------

The following will not solve it:

- changing PPO hyperparameters
- changing runner settings
- changing reward scales alone
- changing terrain settings
- changing Isaac Gym imports

Those are unrelated to the dimensional mismatch.

15.7 What would solve this problem
----------------------------------

There are only a few valid directions:

1. Use a real G1 retargeted motion file
   The cleanest solution.
   The `.npy` file should have one frame per row and 29 joint targets per frame.

2. Retarget the existing 19-DOF motion to G1
   This means creating a mapping from the 19-joint source representation to the
   29-joint G1 representation.
   This is not automatic in the current code.

3. Change the imitation target to only use a subset of G1 joints
   For example, if your goal is upper-body tracking only, you could:
   - keep lower-body joints outside target_jt tracking,
   - define a smaller target vector,
   - add indexing/masking logic in `_reward_target_jt()` and in observation assembly.

4. Pad the 19-DOF target to 29 DOFs with a defined mapping
   This is only valid if you know exactly which 19 joints correspond to which 29
   G1 joints and what the missing 10 joints should do.
   Blind zero-padding would run, but it would not be semantically correct unless
   you define the mapping carefully.

15.8 Why this matters for your intended design
----------------------------------------------

You said your long-term goal is:
- lower body for stability
- upper body for imitation learning

This blocker is exactly where that decision becomes concrete.

Right now the code assumes:
- full target_jt for all 29 DOFs

But your current file only gives:
- 19 target values

So before continuing, you will eventually need to decide:

- full-body imitation on G1 using 29-DOF retargeted data,
or
- partial-body imitation using explicit joint subset logic.

15.9 Short version
------------------

The current trajectory file is not in G1 joint space.

The environment now works well enough to ask for a 29-DOF G1 motion target,
but the file you give it still contains only 19 values per frame.

So the remaining problem is no longer framework migration.
It is trajectory representation mismatch.
